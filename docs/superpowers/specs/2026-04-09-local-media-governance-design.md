# 本地离线媒体治理系统设计

- 日期: 2026-04-09
- 状态: Draft
- 目标平台: macOS 单机
- 目标数据规模: 5-6 块硬盘，总量约 10TB，图片和视频都很多

## 1. 背景与目标

用户有多块机械硬盘和固态硬盘，内部存储了大量图片、影片及其他资源。当前主要问题：

- 资源分散在多块盘中，缺乏统一索引
- 存在大量完全重复或内容相同但质量不同的文件
- 资源质量参差不齐，人工筛查成本过高
- 文件名和目录结构并不能可靠反映内容

本项目目标不是“自动删除文件”，而是构建一套长期可用的本地离线治理系统：

- 统一扫描多块硬盘中的图片和视频
- 通过 AI 理解内容，生成直白明确的标签和描述
- 发现完全重复、同内容不同版本、同一人、同一系列等候选关系
- 提供类似照片管理器的 Web 工作台，支持多维筛选和人工决策
- 最终决策由人做，系统只负责理解、组织、建议和记录

## 2. 非目标

第一版明确不做：

- 自动永久删除文件
- 云端模型推理或上传敏感素材
- 原生 macOS 客户端
- “影视库播放器”产品能力
- 复杂分布式部署

## 3. 核心原则

### 3.1 本地离线优先

所有核心能力在本地完成：

- 文件扫描
- 缩略图和关键帧提取
- 哈希与向量生成
- AI 标签和描述生成
- 聚类和候选建议
- 元数据与索引存储

### 3.2 AI 负责理解，人负责裁决

系统输出：

- 内容标签
- 质量标签
- 描述摘要
- 聚类候选
- 保留/清理建议

系统不直接输出强删除动作。所有清理动作先进入候选态，由用户确认。

### 3.3 不把内容标签做成封闭词表

标签体系采用三层结构：

- 固定结构维度
- AI 自动生成的开放内容标签
- 人工补充标签

其中：

- 管理标签必须规范化
- 质量标签适合半规范化
- 内容标签允许 AI 直白明确地生成，不做含糊 euphemism

### 3.4 图片和视频统一纳入主流程

第一版必须完整支持图片和视频，不采用“图片先做、视频后补”的路线。

## 4. 用户体验目标

目标形态是“照片管理器式 Web 工作台”，重点能力：

- 网格浏览图片和视频封面
- 按名称、路径、标签、描述、状态筛选
- 查看同一人、同一系列、同内容三类候选组
- 查看 AI 理解结果和推荐理由
- 做出 `保留 / 待清理 / 忽略建议 / 收藏` 等人工决策

不是表格型运维后台，也不是命令行一次性清理工具。

## 5. 技术选型

## 5.1 总体架构

- 前端: Web UI
- 主服务: Go
- AI Worker: Python sidecar
- 数据库: PostgreSQL
- 向量检索: pgvector
- 全文检索: PostgreSQL tsvector
- 媒体处理: ffmpeg / ffprobe

## 5.2 为什么不做原生客户端

当前优先级是验证核心链路：

- 扫描
- 理解
- 聚类
- 审核

原生壳不是当前价值中心。Web UI 更适合快速迭代高密度媒体审核界面。

## 5.3 AI 推理后端建议

项目默认后端建议：

- 默认后端: Ollama
- 本机试模型与人工验证: LM Studio
- Worker 代码结构: provider adapter，可兼容多个推理后端

原因：

- Ollama 更适合作为长期运行的应用依赖
- LM Studio 更适合本机快速试模型、看输出和调 prompt
- 代码层不应与单一推理后端绑定

### 5.3.1 Worker 结构

第一版采用：

- `Python sidecar`
- `单 worker 进程`
- `多 provider adapter`
- `统一任务协议`

不采用：

- 每种能力拆成独立 worker
- 直接把 Worker 与单一 provider 绑定

这样可以同时满足：

- 第一版实现简单
- provider 可替换
- 后续可增加 MLX 或其他本地后端

### 5.3.2 Worker 职责边界

Worker 负责：

- 接收图片或视频代表帧
- 生成内容标签
- 生成描述摘要
- 生成敏感标签
- 生成质量辅助判断
- 输出统一结构化 JSON

Worker 不负责：

- 最终聚类裁决
- 最终删除动作
- 主数据库业务逻辑

也就是说，Worker 负责“理解”，主服务负责“落库、规范化、聚类和状态管理”。

### 5.3.3 统一任务协议

第一版统一成一个主任务：

- `understand_media`

输入建议包含：

- `file_id`
- `media_type`
- `file_path`
- `thumbnail_paths` 或 `frame_paths`
- `context`

其中 `context` 可包含：

- 是否允许直白敏感标签
- 输出语言
- 最大标签数
- 标签粒度偏好

输出结构建议包含：

- `raw_tags`
- `canonical_candidates`
- `summary`
- `sensitive_tags`
- `quality_hints`
- `confidence`
- `provider`
- `model`
- `raw_response`

### 5.3.4 Provider Adapter 最小接口

第一版 provider adapter 只定义最小能力：

- `health_check()`
- `supports_vision()`
- `generate_understanding(input)`

避免第一版把 provider 抽象设计得过重。

### 5.3.5 降级策略

默认优先级：

1. `Ollama`
2. `LM Studio`

当视觉模型不可用时：

- 系统仍可完成扫描、元数据提取、embedding、聚类候选生成
- 仅缺失 AI 标签和描述
- 不允许因为 VLM 不可用而阻断整条链路

## 5.4 Mac 单机模型建议

设备条件：

- 32GB Mac mini

推荐模型：

- 主推荐: Qwen3-VL-8B
- 备选: Qwen2.5-VL-7B
- 高速粗标备选: Qwen3-VL-4B

不建议第一版默认使用过大的模型。第一版重点是跑通整套离线流水线，而不是追求单次最强推理效果。

## 5.5 Embedding 策略

第一版不采用“一个 embedding 模型打天下”的方案，而采用双路线：

- `通用视觉 embedding`
- `人脸 embedding`

文本 embedding 不是第一版阻塞项，可在后续增强搜索时补充。

### 5.5.1 通用视觉 embedding

职责：

- 图片相似检索
- 视频关键帧相似检索
- `same_content`
- `same_series`

要求：

- 图片和视频关键帧尽量共用同一向量空间
- 模型足够轻，支持本地批处理
- 能稳定支撑 pgvector 相似检索

### 5.5.2 人脸 embedding

职责：

- `same_person`

启用条件：

- 检测到清晰可用的人脸
- 人脸尺寸达到最低阈值

若素材不满足条件：

- 不强行做 `same_person` 判定
- 仍然参与 `same_content` 和 `same_series`

### 5.5.3 召回策略

`same_person` 第一版采用：

- `高召回优先`
- `人工复核兜底`

也就是说：

- 宁可产生更多候选
- 也不在第一版过早做强判定和过度过滤

这与本项目“AI 负责理解和组织，人负责最终裁决”的总体原则一致。

### 5.5.4 聚类责任划分

第一版明确：

- VLM 负责内容理解、标签和描述
- embedding 负责相似度计算和聚类候选召回
- 主服务负责阈值规则、候选组合并和状态管理

## 6. 系统模块设计

系统拆为 5 个主要模块。

### 6.1 扫描器

职责：

- 遍历所有接入硬盘
- 发现新增、变更、缺失文件
- 记录盘状态

要求：

- 支持首次全量扫描
- 支持后续增量更新
- 卷离线时保留历史，不把文件立即认定为已删除

### 6.2 提取器

职责：

- 提取图片和视频的基础元数据
- 生成缩略图、封面和关键帧
- 计算哈希和基础视觉特征

图片提取项：

- 宽高
- 格式
- EXIF
- 缩略图
- 文件哈希
- 感知哈希
- 视觉 embedding

视频提取项：

- 时长
- 分辨率
- 编码
- 码率
- 封面
- 关键帧
- 关键帧感知哈希
- 关键帧 embedding

### 6.3 理解与标注引擎

职责：

- 调用本地视觉模型
- 生成内容标签和描述
- 生成管理标签和质量标签

输出：

- `内容标签`
- `结构化属性`
- `描述摘要`
- `推荐理由`

### 6.4 聚类引擎

聚类类型必须拆开，不混用：

- `same_content`: 同内容、重复、不同转码、不同尺寸版本
- `same_person`: 同一人候选组
- `same_series`: 同一系列、同一场景、同一拍摄批次

### 6.4.1 聚类误判优先级

当前用户最不能接受的误判是：

- `不同系列被错误合并成同一系列`

因此第一版聚类策略应满足：

- `same_series` 保守
- `same_person` 高召回
- `same_content` 中等偏保守

具体含义：

- `same_series` 宁可拆散，也不要过早合并不同系列
- `same_person` 宁可产生更多候选，由人工复核
- `same_content` 需避免把相似但不相同的内容强行并为同内容

### 6.5 审核工作台

职责：

- 浏览资源
- 筛选资源
- 查看 AI 结果
- 处理聚类候选
- 记录人工裁决

## 7. 标签设计

## 7.1 标签层次

### A. 内容标签

由 AI 自动生成，允许直白明确，不做封闭词表。

示例：

- `AV`
- `自拍做爱`
- `单人写真`
- `泳装写真`
- `情侣性爱`
- `室内床上`
- `手机竖屏自拍`
- `聊天记录截图`

### B. 描述摘要

每个文件输出一句简短自然语言说明。

示例：

- `单人室内写真，竖版半身，清晰度高`
- `手机自拍性爱视频，室内，单镜头，时长 18 分钟`

### C. 结构化属性

供稳定筛选，不完全依赖模型自由表达。

例如：

- `媒体类型`
- `主体数量`
- `拍摄形态`
- `横竖屏`
- `清晰度`

### D. 管理与质量标签

用于治理而非表达内容：

- `完全重复`
- `疑似重复`
- `低清晰度`
- `压缩明显`
- `待审核`
- `可优先清理`

### E. 人工标签

用户手工添加：

- `收藏`
- `保留`
- `待删`
- `忽略`

## 7.2 标签落库结构

第一版标签结果分三层持久化：

- `raw_tags`
- `canonical_tags`
- `structured_attributes`

### 7.2.1 raw_tags

含义：

- 保留模型原始表达
- 允许直白敏感内容
- 不强制词表收敛

用途：

- 原始内容理解留档
- 搜索增强
- 供后续人工判断和规范化修正参考

### 7.2.2 canonical_tags

含义：

- 面向筛选和统计的规范标签
- 采用宽松归一化，不做封闭词表

原则：

- 只合并明显同义、近乎等义的表达
- 合并前提是不改变原意
- 不为了整洁而牺牲语义细节

示例：

- `手机自拍` 与 `手机自拍视频` 可合并
- `竖屏自拍` 与 `手机竖屏自拍` 可按规则保留更完整表达或合并到同一 canonical 形式
- `AV` 与 `成人视频` 若在当前上下文中确属同义，可归并

但以下情况不应强合并：

- `自拍做爱` 与 `情侣性爱`
- `单人写真` 与 `自拍`
- `剧情片段` 与 `正片`

### 7.2.3 structured_attributes

含义：

- 作为稳定筛选维度存在
- 不依赖自由标签表达

示例字段：

- `media_type`
- `subject_count`
- `capture_type`
- `orientation`
- `resolution_tier`
- `quality_tier`
- `has_face`
- `is_sensitive`

## 7.3 规范化策略

第一版采用：

- `宽松归一化`
- `不改变原意`
- `保留原始标签`

执行顺序：

1. 模型输出 `raw_tags`
2. 规范化器生成 `canonical_tags`
3. 规则层补充或修正 `structured_attributes`
4. 三层结果同时保留

### 7.3.1 归一化边界

允许：

- 同义词归并
- 中英文等义表达归并
- 轻微词序差异归并
- 明显冗余修饰词裁剪

不允许：

- 改变内容主体
- 改变行为语义
- 改变场景语义
- 把上下位概念强行视作同义

### 7.3.2 筛选优先级

工作台筛选优先使用：

1. `structured_attributes`
2. `canonical_tags`
3. `raw_tags`

原因：

- 结构化字段最稳定
- canonical 标签次之
- raw 标签表达力最强，但噪声也最大

## 7.4 标签命名空间

建议为标签加命名空间，避免混乱：

- `content:*`
- `scene:*`
- `quality:*`
- `management:*`
- `sensitive:*`
- `person:*`

## 8. 图片与视频理解链路

## 8.1 图片

流程：

1. 读取基础元数据
2. 生成缩略图
3. 计算文件哈希
4. 计算感知哈希
5. 生成视觉 embedding
6. 调用本地视觉模型输出标签和描述

## 8.2 视频

视频不整段送模型，采用“抽样理解”。

流程：

1. 提取基础元数据
2. 生成封面
3. 抽取关键帧
4. 对关键帧生成感知哈希和 embedding
5. 从关键帧中选代表帧送入视觉模型
6. 汇总成视频级标签和描述

## 8.3 保留建议

系统不直接删除，只输出候选建议：

- 同内容下优先保留分辨率更高版本
- 分辨率接近时优先保留码率更高、压缩痕迹更少的版本
- 命中 `截图 / 缩略图 / 低清搬运 / 重复转码` 的版本更容易进入清理候选

## 9. 关键帧策略 v1

关键帧数量不固定，按时长动态分档。

第一版偏好：

- `完整理解优先`
- 相比“省资源”，更愿意多抽一些帧保证语义覆盖
- 但仍需设置合理上限，避免长视频任务无限膨胀

### 9.1 抽帧数量建议

- `0-30 秒`: 3 帧
- `30 秒-5 分钟`: 5 帧
- `5-20 分钟`: 8 帧
- `20-60 分钟`: 12 帧
- `60 分钟以上`: 16 帧

在“完整理解优先”的策略下，第一版将上限上调为：

- `0-30 秒`: 4 帧
- `30 秒-5 分钟`: 6 帧
- `5-20 分钟`: 10 帧
- `20-60 分钟`: 16 帧
- `60 分钟以上`: 24 帧

其中：

- `索引帧` 与 `理解帧` 使用同一候选池
- 最终送入 VLM 的帧数量仍需受控，不等于全部抽帧数

### 9.2 采样策略

采用混合策略：

- 70% 均匀采样
- 30% 变化点补采样

在长视频场景下，先按时间把视频切成多个区段，再在每个区段内部做混合采样，避免代表帧过度集中在少数片段。

推荐流程：

1. 跳过片头前 3-5 秒和片尾后 3-5 秒
2. 按时长切分为若干区段
3. 每段先做均匀采样
4. 再在段内用帧差或镜头变化指标补采样
5. 将所有候选帧去重后进入后续流程

### 9.3 过滤规则

尽量跳过：

- 开头前几秒和结尾后几秒
- 纯黑帧、纯白帧
- 明显转场模糊帧
- 重复度很高的相邻帧

同时增加以下过滤：

- 大面积字幕页或片头片尾页
- 长时间静止且信息量低的帧
- 同一区段内内容几乎不变的重复帧

过滤原则：

- 不追求“最少帧”
- 追求“尽量多保留有信息量的代表帧”
- 在保留信息覆盖的前提下再做去重

### 9.4 两层关键帧

建议把关键帧拆成两类：

- `索引帧`: 给去重、相似检索和聚类使用，数量少
- `理解帧`: 给视觉大模型打标签和生成描述使用，数量略多

对于 32GB Mac mini，建议首版控制在：

- `索引帧`: 默认 6-12，超长视频最多 16
- `理解帧`: 默认 4-8，超长视频最多 10

### 9.5 帧的用途分层

不是所有抽出的帧都送给 VLM。

建议分三层：

1. `候选帧池`
由抽帧器生成，覆盖尽量完整

2. `索引帧`
从候选帧池中选出，进入感知哈希与 embedding 流程

3. `理解帧`
从候选帧池中选出最具代表性和信息量的帧，送入 VLM

选择原则：

- 优先保留不同区段的代表帧
- 避免 VLM 输入过多相似帧
- 避免全部代表帧都落在同一个场景

### 9.6 视频级理解合成

视频标签和描述不是来自单帧，而是来自多帧汇总。

建议流程：

1. 对每个理解帧生成单帧标签与描述
2. 合并重复标签
3. 保留跨帧重复出现的高置信标签
4. 将局部标签汇总成视频级标签和摘要

这样可以降低“单帧偶然性”对视频理解结果的影响。

## 9.7 质量评分规则 v1

第一版质量评分采用混合模型：

- `客观技术指标`: 70%
- `AI 主观视觉判断`: 30%

目标不是给出绝对美学分，而是服务于以下治理动作：

- 同内容候选中的保留版本推荐
- `trash_candidate` 候选生成
- 聚类中的 `best_quality` 成员选择

### 9.7.1 总体原则

- 规则主导，AI 辅助
- 尽量优先使用可解释的客观指标
- AI 判断只作为补充，不单独决定删除或保留
- 评分结果应可追溯到具体证据

### 9.7.2 图片质量评分

图片客观指标建议包含：

- 分辨率
- 长边尺寸
- 文件大小与尺寸匹配度
- 压缩痕迹
- 模糊程度
- 裁切完整度
- EXIF 或原始格式信息
- 是否为缩略图、截图、二次转存版本

图片 AI 辅助判断建议包含：

- 主体是否清晰
- 构图是否完整
- 画面是否存在明显压缩损伤
- 是否存在严重遮挡或噪声
- 是否属于明显低价值图，如无意义截图、缩略海报、低质量搬运图

### 9.7.3 视频质量评分

视频客观指标建议包含：

- 分辨率
- 码率
- 时长
- 编码格式
- 帧率
- 黑边情况
- 音视频轨道完整性
- 是否存在硬字幕
- 是否属于明显低清转码版本

视频 AI 辅助判断建议包含：

- 关键帧主体是否清晰
- 画面是否明显模糊或过度压缩
- 是否存在严重噪点、马赛克或转码损伤
- 代表帧是否稳定呈现低质量观感

### 9.7.4 评分输出结构

建议同时输出：

- `quality_score`
- `quality_tier`
- `quality_flags`
- `quality_reasons`

其中：

- `quality_score`: 数值分
- `quality_tier`: 例如 `high | medium | low`
- `quality_flags`: 例如 `low_resolution`, `heavy_compression`, `hard_subtitle`
- `quality_reasons`: 便于前端展示和人工理解的解释文本

### 9.7.5 在治理中的使用方式

质量评分主要用于：

- 同内容组内排序
- 推荐保留版本
- 降低低质量重复资源的优先级

质量评分不应单独用于：

- 直接删除文件
- 直接否决人工保留意图

## 10. 数据库设计 v1

第一版围绕“资产库”建模，而不是围绕目录树建模。

数据库设计原则：

- `尽量规范化`
- `允许适度冗余`

解释：

- 核心实体关系优先保持清晰、稳定、可追溯
- 但为了审核工作台查询、筛选、排序和搜索性能，可引入少量冗余字段或派生表

第一版主实体拆分原则：

- `标签` 独立建模
- `文件` 与 `标签` 通过关联表连接
- `描述`、`质量评分`、部分 AI 分析结果可归入统一的 `analysis_result` 表

这样做的原因：

- 标签体系需要长期维护、去重、归一化和复用
- 描述与质量评分更像一次或多次分析产物，适合集中管理

`analysis_result` 采用：

- `多版本存储`
- `当前生效版本指针`
- `按 analysis_type 区分`

不采用“每个文件只保留一份最新分析结果”的方案。

原因：

- 后续可能切换模型
- 可能调整 prompt 或规则
- 需要支持重跑分析而不丢失旧结果

`analysis_result` 与 `标签` 的边界明确如下：

- `analysis_result` 不负责维护正式标签关系
- `tags` 与 `file_tags` 负责正式标签体系
- `analysis_result` 可保存原始标签建议或证据，但不直接替代标签关联表

推荐做法：

- `analysis_result.structured_attributes` 使用 `jsonb`
- `analysis_result` 保存 `summary / caption / quality / attributes / raw_model_output`
- 标签定义与标签绑定走独立表，便于长期维护、归一化和人工修正

建议的 `analysis_type`：

- `understanding`
- `quality`
- `search_doc`

第一版可以共用同一张 `analysis_results` 表，但必须带 `analysis_type` 字段。

`raw_model_output` 策略：

- 第一版尽量完整保留原始模型输出
- 同时需要做大小保护，避免单条记录无限膨胀

原因：

- 便于排查模型行为
- 便于后续 prompt 调整
- 便于分析 AI 标签与最终规范标签之间的差异

## 11. 任务队列设计 v1

第一版只使用 PostgreSQL 表作为队列，不引入外部 MQ。

调度原则：

- `简单可靠`
- `幂等优先`
- `允许重复跑`

不追求第一版就实现极致复杂的任务去重。

原因：

- 本地离线扫描系统更需要稳定恢复能力
- 任务重复执行通常比“卡死在半状态”更安全
- 多数分析任务可设计成幂等写入或幂等替换

### 11.1 任务表 `jobs`

字段：

- `id`
- `job_type`
- `priority`
- `status`
- `target_type`
- `target_id`
- `payload jsonb`
- `attempt_count`
- `max_attempts`
- `lease_owner`
- `lease_expires_at`
- `scheduled_at`
- `started_at`
- `finished_at`
- `last_error`
- `created_at`
- `updated_at`

`status`：

- `pending`
- `leased`
- `running`
- `succeeded`
- `failed`
- `dead`

### 11.1.1 lease 与重试策略

第一版建议：

- worker 拉取任务后先获取 `lease`
- lease 到期未续约时，任务可重新被调度
- 失败任务按 `attempt_count` 自动重试
- 超过 `max_attempts` 后进入 `dead`

建议默认值：

- 普通任务 lease: `10 分钟`
- 长任务支持续租
- `max_attempts`: `3`

同时要求：

- `last_error` 必须记录最近一次失败原因
- 重试不应清空历史错误信息

### 11.1.2 dead job 处理

第一版要求支持：

- `dead job` 的人工重试

原因：

- 本地模型可能暂时不可用
- 磁盘可能临时未挂载
- ffmpeg 或推理环境可能存在瞬时故障

因此 `dead` 表示“自动重试已放弃”，而不是“永久不可恢复”。

### 11.2 任务类型

- `scan_volume`
- `index_file`
- `extract_image_features`
- `extract_video_features`
- `embed_image`
- `embed_video_frames`
- `infer_tags`
- `infer_summary`
- `cluster_same_content`
- `cluster_same_person`
- `cluster_same_series`
- `recompute_search_doc`

其中：

- `infer_tags` 和 `infer_summary` 可由同一个 `understand_media` worker 请求完成
- 是否拆成两个 job 由主服务调度层控制，不能影响 worker 协议稳定性

### 11.3 队列执行策略

- `scan / index / extract` 可并发
- `infer_*` 严格限流
- `cluster_*` 采用批任务，不在单文件写入后立即全库重算

第一版幂等设计建议：

- `scan_volume` 可重跑
- `index_file` 可重跑
- `extract_*` 可重跑
- `embed_*` 可重跑
- `infer_*` 可重跑
- `recompute_search_doc` 可重跑

对于 `cluster_*`：

- 第一版采用“可整体重算”思路
- 不追求一开始就实现非常细粒度的增量维护
- 聚类结果允许通过批任务重新生成和替换

## 12. 文件处理顺序 v1

单个文件进入系统后的顺序：

1. `scan_volume`
2. `index_file`
3. `extract_image_features` 或 `extract_video_features`
4. `embed_*`
5. `infer_tags`
6. `infer_summary`
7. `cluster_same_content`
8. `cluster_same_person`
9. `cluster_same_series`
10. `recompute_search_doc`

## 13. Mac mini 并发建议

针对 32GB Mac mini，第一版建议如下：

- `scan/index`: 4
- `extract`: 2
- `embed`: 2
- `infer_tags / infer_summary`: 1
- `cluster`: 串行批处理

重点：

- 视频只抽关键帧，不整段进模型
- 视觉大模型不承担聚类主任务
- 聚类主要依赖 embedding + pgvector

`macOS 本地离线媒体治理系统`

它不是简单的去重脚本，而是一个围绕“内容理解、标签生成、聚类候选、人工审核”构建的长期工具。

第一版完全可以在当前设备条件下跑通，但前提是坚持以下边界：

## 22. Web 工作台 IA v1

第一版首页不直接展示“全部资源流”，而是采用：

- `数据面板首页`
- `待处理候选优先`

也就是说，用户打开系统后首先看到的是当前需要处理的治理工作，而不是全量资源瀑布流。

### 22.1 首页目标

首页需要回答的问题：

- 当前总共有多少资源
- 哪些候选最值得优先处理
- 哪些扫描或分析任务还在进行
- 最近有哪些新增变化

### 22.2 首页建议模块

第一版首页建议包含以下面板：

1. `资源总览`

- 图片总数
- 视频总数
- 总占用空间
- 在线 volume 数量

2. `待处理候选`

- 完全重复候选数
- 同内容候选数
- 同一人候选数
- 同一系列候选数
- `trash_candidate` 数量

3. `任务状态`

- 扫描中任务数
- 分析中任务数
- 失败任务数
- 最近完成任务数

4. `近期变化`

- 最近新增文件数
- 最近移动/重命名文件数
- 最近进入待删候选的文件数

5. `快捷入口`

- 进入重复候选审核
- 进入同一人候选审核
- 进入同一系列候选审核
- 进入待删审核
- 进入全部资源浏览

### 22.3 首页定位

首页本质上是：

- `治理驾驶舱`

而不是：

- 图库首页
- 普通资源浏览首页

### 22.4 一级导航建议

第一版建议一级导航为：

1. `首页`
2. `候选审核`
3. `资源浏览`
4. `任务中心`
5. `设置`

其中：

- `候选审核` 作为独立一级导航存在
- `资源浏览` 承担全量筛选与浏览任务
- 首页主要用于总览和分发，不承担重度审核操作

### 22.5 候选审核信息结构

`候选审核` 页面建议再分为 4 个子视图：

1. `同内容`
2. `同一人`
3. `同一系列`
4. `待删候选`

这样做的原因：

- 四类候选的语义不同
- 审核动作不同
- 混在一个列表里会增加认知负担

其中：

- `待删候选` 归属于 `候选审核`
- 不单独升级为一级导航

### 22.6 候选审核布局

第一版候选审核采用：

- `左侧候选组列表`
- `右侧当前组详情`

不采用“所有文件平铺”的审核方式。

原因：

- 用户审核的是“候选关系”
- 需要保留组内比较上下文
- 更适合高密度治理工作流

### 22.7 候选组详情建议展示内容

当前组详情建议包含：

- 组类型
- 组置信度
- 组内文件数量
- 代表封面
- 每个成员的缩略图或视频封面
- 名称与路径
- 关键标签
- 描述摘要
- 质量评分
- 推荐理由
- 推荐保留项或最佳质量项

### 22.8 候选审核操作

候选组内建议支持：

- 保留某个文件
- 标记某个文件为 `trash_candidate`
- 忽略某个成员
- 忽略整个候选组
- 确认该组关系有效
- 手动拆分或移出成员（后续增强可补）

### 22.9 资源浏览页

`资源浏览` 页承担：

- 全量资源浏览
- 搜索
- 多维筛选
- 单文件详情查看

展示形态建议：

- 照片管理器式网格布局

媒体类型默认策略：

- 默认混合浏览
- 提供 `全部 / 图片 / 视频` 切换

这样既符合统一资产视角，也允许用户快速聚焦某一类媒体。

### 22.10 资源浏览筛选策略

第一版筛选风格定为：

- `专业型`

不刻意压缩筛选维度，以满足深度治理和高密度检索需求。

建议支持的筛选维度：

- 名称
- 路径
- 标签
- 媒体类型
- 文件状态
- 质量等级
- 是否敏感
- 是否有人脸
- 候选状态
- 分辨率区间
- 时长区间
- 文件大小区间
- volume
- 最近新增
- 最近变动

其中 `候选状态` 例如：

- 是否属于 `same_person`
- 是否属于 `same_series`
- 是否属于 `same_content`
- 是否属于 `trash_candidate`

### 22.11 单文件详情交互

第一版单文件详情优先采用：

- `右侧详情抽屉`

不要求第一版就建设复杂的独立详情页。

详情抽屉建议展示：

- 预览图或视频封面
- 基础元数据
- 标签
- 描述摘要
- 质量评分
- 所属候选组
- 历史路径
- 主要操作按钮

这样更适合：

- 快速浏览
- 快速审核
- 在资源流中保持上下文不丢失

## 23. 可观测性与恢复 v1

第一版要求具备：

- 独立 `任务中心`
- 可见的扫描/分析进度
- 清晰的失败原因
- 人工重试入口

但不建设过度复杂的运维后台。

### 23.1 任务中心定位

任务中心用于回答：

- 当前系统正在做什么
- 哪些任务排队中
- 哪些任务失败了
- 是否需要人工干预

### 23.2 任务中心默认结构

第一版任务中心默认采用：

- `任务列表`
- `支持按 volume 过滤`

而不是默认按 volume 或扫描批次分组展示。

建议至少包含 4 个视图：

1. `进行中`
2. `等待中`
3. `失败 / dead`
4. `最近完成`

### 23.3 任务列表展示字段

每条任务建议至少展示：

- `job_type`
- `status`
- `target`
- `progress`
- `attempt_count / max_attempts`
- `lease` 状态
- `started_at`
- `duration`
- `last_error`
- `worker / provider`

对于失败或 `dead` 任务，需提供：

- `重试` 入口

### 23.4 进度展示策略

第一版采用两层进度策略：

1. `百分比进度`
适用于可以明确估算处理进度的任务

2. `阶段状态`
适用于难以准确估算百分比的任务

典型阶段可包括：

- `扫描中`
- `抽帧中`
- `embedding 中`
- `推理中`
- `聚类中`

原则：

- 能给百分比就展示百分比
- 给不了百分比时，至少要有明确阶段状态

## 20. 扫描与增量更新机制 v1

第一期扫描策略定为：

- `定时重扫`
- `手动触发扫描`

第二期再补：

- `文件系统事件监听`
- `尽量实时同步`

### 20.1 第一期开工原则

第一版优先目标是：

- 稳定
- 可恢复
- 易解释

因此不把文件系统事件监听作为第一期阻塞项。

### 20.2 第一版扫描方式

支持两种入口：

1. `手动扫描`
用户主动选择对某个卷或全部卷发起扫描

2. `定时扫描`
系统按固定周期触发重扫

建议：

- 卷级扫描
- 支持全量扫描和增量重扫共用同一主流程

### 20.3 增量识别方式

第一版不依赖实时事件流，而依赖重扫时比对：

- `volume_id`
- `abs_path`
- `size_bytes`
- `mtime`
- `quick_hash`

必要时再补 `sha256` 复核。

这意味着第一版的“增量”本质是：

- 周期性重新发现变化
- 而不是事件驱动即时发现变化

### 20.4 路径变化与迁移识别

重扫时若发现：

- 某旧路径消失
- 某新路径出现
- 二者在 `size / quick_hash / sha256` 上高度一致

则优先判断为：

- `迁移 / 重命名`

而不是：

- `删除 + 新增`

这样可以保留：

- 标签
- 聚类关系
- 人工决策记录

这同样适用于：

- 同一卷内手动移动文件夹
- 不同目录间手动移动文件
- 不同 volume 之间手动迁移文件

第一版是否能监测到，取决于：

- 是否发生了后续重扫
- 文件的 `size / quick_hash / sha256` 是否足够匹配

也就是说：

- `可以识别`
- 但第一版是 `扫描后识别`
- 不是 `实时监听后立刻识别`

### 20.4.1 迁移识别优先级

第一版建议按以下优先级判断迁移：

1. `sha256` 一致
2. `quick_hash + size_bytes` 一致
3. 必要时再结合 `mtime`、扩展名、媒体元数据辅助判断

当文件跨 volume 移动时：

- 只要指纹足够一致，仍优先视为同一文件迁移
- 应尽量继承原有标签、分析结果、聚类关系和人工状态

### 20.4.2 已知边界

第一版以下场景仍可能不稳定：

- 移动后文件内容被重新转码或再次压缩
- quick hash 相同但内容实际已变化
- 路径变化与内容变化同时发生，且缺少可靠哈希

对于这类情况：

- 可降级为“删除 + 新增”
- 或进入“疑似迁移待确认”候选状态

### 20.5 卷离线处理

当某块盘在扫描时不可访问：

- 将卷标记为 `offline`
- 不立即把文件标记为删除

文件层仅在满足更强条件时才进入 `missing`：

- 卷曾经在线
- 本次目标扫描应包含该卷
- 该卷当前不可访问或重扫结果确认文件缺失

第一版建议：

- 先标记 `volume offline`
- 再根据连续多次扫描结果决定文件是否进入 `missing`

### 20.6 定时策略建议

第一版建议支持简单配置：

- `禁用定时扫描`
- `每日一次`
- `每 6 小时一次`
- `每小时一次`

默认建议：

- `每日一次`

### 20.7 第二期方向

第二期增加文件系统事件监听后：

- 实时事件用于快速发现新增、修改、重命名
- 定时重扫仍然保留，作为最终一致性兜底

也就是说，事件监听不是替代重扫，而是增强重扫。

## 21. 回收区与删除策略 v1

第一版删除策略定为：

- 平时只做 `逻辑标记`
- 删除与否由人二次审核
- 页面支持显式删除动作
- 删除时将文件移动到 `macOS 废纸篓`

### 21.1 默认状态

系统默认不直接删除文件。

AI 或规则层只能输出：

- `trash_candidate`

这表示：

- 文件进入待清理候选
- 仍然保留在原位置
- 需用户在工作台中再次确认

### 21.2 人工二次审核

用户在工作台中可对候选执行：

- `保留`
- `忽略建议`
- `删除`

其中：

- `保留` 表示明确保留该文件
- `忽略建议` 表示当前不采纳系统建议，但不一定永久豁免
- `删除` 表示执行真实文件删除动作

### 21.3 第一版物理删除方式

当用户点击删除后：

- 不做应用内自建回收区
- 直接将文件移动到 `macOS 废纸篓`

原因：

- 符合用户已有使用习惯
- 便于后续人工恢复
- 避免第一版自行维护跨盘隔离目录和二次回收逻辑

### 21.4 状态与记录

建议记录以下状态：

- `trash_candidate`
- `kept`
- `ignored`
- `deleted_to_trash`

其中 `deleted_to_trash` 表示：

- 应用侧已执行删除动作
- 文件已尝试移动到系统废纸篓

### 21.5 异常处理

若移动到废纸篓失败：

- 不应默默吞掉错误
- 页面需要提示失败原因
- 文件状态保持不变

### 21.6 后续扩展方向

若未来需要更强治理能力，可在后续版本增加：

- 应用内隔离区
- 批量删除审核页
- 删除恢复记录
- 不同卷的清理策略

## 18. 聚类规则增补 v1

### 18.1 same_content

`same_content` 只在以下情况成立：

1. `完全重复`
2. `同一内容不同版本`

其中：

- `完全重复` 由文件哈希直接判定
- `同一内容不同版本` 需要多个信号同时满足，例如：
  - 视觉 embedding 足够接近
  - 图片 pHash 或视频关键帧相似度足够高
  - 视频时长接近，或图片仅存在尺寸/编码差异

第一版不把“视频截取片段”直接并入 `same_content` 组。

对于视频片段，仅输出：

- `疑似来自同一原视频`

也就是说：

- 片段与完整原片建立“关联候选”
- 不直接并入同一个 `same_content` 组

这样可避免 `same_content` 被片段污染，也更利于后续人工审核。

### 18.2 same_person

`same_person` 第一版目标是：

- `高召回`
- `人工复核收口`

主依据：

- 人脸 embedding

辅助依据：

- 人脸数量
- 构图相似度
- 整体视觉 embedding
- 场景和服装相似度

但这些辅助依据不能替代主依据。

第一版输出形式：

- `same_person` 候选组
- 每个成员带 `score`
- 不直接输出“确定同一人”的强结论

对于以下情况，第一版也尽量纳入候选，但显著降低置信度：

- 侧脸
- 遮挡
- 远景
- 清晰度较差
- 多人素材

处理原则：

- 只要存在可疑人像线索，就优先召回
- 噪声交给人工复核处理
- 不因为缺少完美正脸而过早放弃召回

### 18.3 same_series

`same_series` 表示：

- 同一套素材
- 同一次拍摄
- 同一批内容
- 同一场景下的连续资源

它不是按预置类别归类，而是一种“同组关系”。

第一版总体策略：

- `保守建组`
- `粒度偏细`
- `宁可拆成多个小组，也不要并成一个大杂组`

#### 18.3.1 same_series 的正向证据

进入 `same_series` 候选时，优先考虑以下证据组合：

- 同一人或主体高度相似
- 场景高度相似
- 背景或布景高度相似
- 服装或造型高度相似
- 构图风格接近
- 文件时间接近
- 路径或目录来源接近
- 图片或关键帧视觉 embedding 接近

#### 18.3.2 硬条件与辅助条件

第一版建议：

- 不设置单一绝对硬条件
- 采用“多证据共同成立”原则

但以下规则明确成立：

- `同一个人` 不能单独推出 `same_series`
- `同一题材` 不能单独推出 `same_series`
- `同一目录` 不能单独推出 `same_series`

也就是说，`same_person` 只能作为强辅助信号，不能直接等价于 `same_series`。

#### 18.3.3 典型应当归为 same_series 的情况

- 同一套写真
- 同一场景下连续自拍
- 同一背景、同一服装、同一拍摄批次的一组图片
- 同一视频中拆出的相邻相关片段
- 同一资源包中的同主题连续内容

#### 18.3.4 典型不应归为 same_series 的情况

- 只是同一个人，但场景明显不同
- 只是同一个人，但服装和背景明显不同
- 只是同一题材，比如都属于写真或都属于 AV
- 只是同一文件夹中的内容，但不属于同一拍摄组
- 只是同一演员参与，但来自不同作品或不同片段

#### 18.3.5 same_series 与 same_person 的关系

推荐交互：

- 系统默认不把 `same_person` 自动提升为 `same_series`
- 工作台可提供“基于 same_person 扩展生成系列候选”的人工触发能力

这样既允许用户更激进地探索候选，又不污染系统默认分组。

#### 18.3.6 视频中的 same_series

视频场景下，`same_series` 更偏向以下含义：

- 同一原视频的相邻内容段
- 同一场景下拍摄的多段自拍视频
- 同一批导入的同主题短视频

但：

- “疑似来自同一原视频”的片段关系优先记为关联候选
- 只有当场景、时间、风格等证据进一步成立时，才进入 `same_series`
