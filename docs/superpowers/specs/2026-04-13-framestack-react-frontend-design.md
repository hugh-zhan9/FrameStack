# FrameStack React 前端重构设计

## 目标

把当前由 Go 直接托管的原生 `HTML + JS + CSS` 工作台，替换为一套可维护的、响应式的 `React + TypeScript` 前端。

本次重构的目标不是改后端，也不是追加新业务协议，而是：

- 保留现有 Go API
- 重建前端工程结构
- 提升文件浏览、预览、聚类审核和 Volume 管理体验
- 让前端具备后续持续迭代的基础

## 非目标

本次不做这些事情：

- 不引入 SSR、Next.js、Remix 等重型框架
- 不重写后端 API
- 不在第一版重构中加入登录、权限、多用户
- 不在第一版中做复杂动画和品牌化营销页面
- 不要求一次性删除旧的 `internal/httpserver/assets/*`

## 方案选择

### 方案 A：继续维护原生静态页面

优点：

- 改动最小
- 不需要引入新前端构建链路

缺点：

- 当前页面复杂度已经超过原生脚本的舒适区
- 状态管理、响应式布局、文件浏览器、审核工作流都会越来越难维护
- 后续继续扩展几乎一定会造成重复重写

不采用。

### 方案 B：`Vite + React + TypeScript`

优点：

- 足够轻
- 组件化和状态管理边界清晰
- 开发体验和调试体验明显更好
- 适合本地工具型工作台
- 可继续由 Go 托管静态资源

缺点：

- 需要补完整前端工程、构建和开发代理
- 需要阶段性维护新旧两套前端入口

采用。

### 方案 C：Next.js / Remix

优点：

- 路由、数据获取、框架能力成熟

缺点：

- 对本地单机工具过重
- 引入大量当前不需要的复杂度
- 和现有 Go API/静态托管组合不自然

不采用。

## 技术设计

### 前端栈

- `Vite`
- `React`
- `TypeScript`

补充约束：

- 不引入 UI 大而全组件库
- 优先用少量自建组件和清晰的 CSS 变量体系
- 只在明确需要时引入轻量依赖

### 工程结构

新增：

- `web/`

建议结构：

```txt
web/
  index.html
  package.json
  tsconfig.json
  vite.config.ts
  src/
    main.tsx
    app/
    pages/
    components/
    features/
    lib/
    styles/
```

### 开发模式

- React dev server 独立运行
- 通过 Vite proxy 转发到 Go API
- Go 继续提供后端接口

这意味着开发阶段可以分别调试：

- 前端布局和交互
- 后端 API 和任务链路

### 生产模式

- `web` 构建输出到 `web/dist`
- Go 在生产模式下优先托管 `web/dist`
- 旧的 `internal/httpserver/assets/*` 保留为过渡实现，直到 React 版稳定

## 页面信息架构

### 1. Dashboard

用途：

- 展示系统状态
- 展示任务摘要
- 展示候选摘要

内容：

- System Status
- Task Summary
- Candidate Summary
- 快速跳转到 `Review` 和 `Library`

### 2. Volumes

用途：

- 管理 volume
- 触发扫描

内容：

- volume 列表
- 添加 volume
- 扫描按钮
- online/offline 状态

第一版约束：

- 先保留路径输入
- 文件夹选择器作为下一步增强点，但结构上要为它预留位置

### 3. Library

用途：

- 作为主文件浏览器
- 重点解决“文件列表看不到、无法预览”的问题

内容：

- 搜索栏
- 多维筛选
- 文件网格 / 列表切换
- 图片和视频预览
- 多选和批量操作
- 详情侧栏

这是本次前端重构的核心页面。

### 4. Review

用途：

- 聚类审核

内容：

- `same_content`
- `same_series`
- `same_person`
- 候选列表
- 详情面板
- 连续审核

### 5. Tasks

用途：

- 查看任务运行状态和失败原因

内容：

- 最近任务
- 失败任务
- 任务时间线

## 响应式布局

### 桌面端

- 左侧：导航与次要列表
- 中部：主内容区
- 右侧：详情侧栏

目标：

- 保持“工作台”结构
- 让文件浏览与详情联动更短

### 平板端

- 双栏布局
- 详情区可切换为抽屉

### 手机端

- 单栏布局
- 详情改为全屏抽屉或独立页面

原则：

- 不要求手机端拥有桌面端全部效率
- 但必须可查看、可筛选、可做核心审核动作

## 关键交互设计

### 文件浏览

- 默认以媒体卡片网格展示
- 每个卡片必须可直接看到预览图
- 视频卡片需显示明确的视频标识
- 支持多选
- 支持批量 `keep / favorite / trash_candidate`

### 文件详情

- 右侧常驻详情面板
- 展示：
  - 基础信息
  - 标签
  - 当前分析
  - embedding 摘要
  - 视频关键帧
  - 所属聚类
  - 人工操作入口

### 聚类审核

- 左侧聚类队列
- 中部或右侧聚类详情
- 支持：
  - 上下切换
  - 连续审核
  - 组级动作

### 刷新策略

- 任务区自动轮询
- 触发扫描、重聚类、重算 embedding 后自动局部刷新
- 尽量减少“手动刷新整个页面”

## 状态管理策略

采用简单、显式的前端状态组织：

- 页面级状态按功能拆分
- API 请求封装统一放到 `lib/` 或 `features/`
- 不引入复杂全局状态框架作为第一步

原则：

- 状态尽量靠近使用位置
- 共享状态只提到必要的页面层
- 避免过早引入重型状态管理

## 视觉方向

前端重构后的视觉方向应从“英文后台面板”转为“中文媒体治理工作台”。

要求：

- 中文主文案
- 信息层级清楚
- 文件预览成为视觉主角
- 不使用过度花哨的品牌化风格
- 优先保证高密度信息下的可读性和操作效率

## 迁移计划

### 阶段 1：建立 React 工程

- 初始化 `web/`
- 接通 Vite
- 接通 TypeScript
- 接通开发代理

### 阶段 2：应用壳与路由

- 建立基础布局
- 建立导航
- 建立页面骨架

### 阶段 3：重做 Library

- 文件浏览
- 文件预览
- 文件详情
- 批量操作

这是第一优先级。

### 阶段 4：重做 Dashboard 和 Volumes

- 仪表盘
- 卷管理
- 扫描入口

### 阶段 5：重做 Review

- 三类聚类审核
- 连续审核工作流

### 阶段 6：重做 Tasks 与收尾

- 任务中心
- 失败态
- 细节 polish

## 验收标准

满足以下条件即可认为前端重构第一版完成：

- React 工程可独立开发和构建
- Go 可托管前端构建产物
- 首页、Volume、Library、Review、Tasks 五个页面可访问
- `Library` 页面可真正预览图片和视频
- 文件列表、详情、聚类审核都具备响应式布局
- 当前原生静态前端不再是主入口

## 风险与控制

### 风险 1：一次性替换过大

控制方式：

- 保持后端 API 不动
- 先重做前端，不同时重写业务协议

### 风险 2：Library 页面复杂度最高

控制方式：

- 把 `Library` 作为第一优先级单独落
- 先做可用，再做 polish

### 风险 3：新旧前端并存造成混乱

控制方式：

- React 前端稳定前保留旧入口
- 稳定后再切主入口

## 当前结论

本次重构采用：

- `Vite + React + TypeScript`
- 保留现有 Go API
- 重建响应式前端工作台
- 以 `Library` 页面为第一优先级

这是一条足够稳、维护成本可控、并且能真正解决当前前端不足的路径。
